# AWS SES メール送信 IAM 権限設定の試行錯誤

## 背景

Lambda から SES v2 (`aws_sdk_sesv2`) でメール送信する機能を実装。ローカルでは動作するが、クラウド（Lambda）からは `AccessDeniedException` が返る問題を調査・解決した。

---

## 試行① `grantSendEmail(identity/surfic.com)` → 失敗

CDK の `EmailIdentity.grantSendEmail` を使い、`surfic.com` ドメイン identity に限定した権限を付与した。

```ts
const identity = aws_ses.EmailIdentity.fromEmailIdentityArn(
    this, 'SesIdentity',
    `arn:aws:ses:${this.region}:${this.account}:identity/surfic.com`,
);
identity.grantSendEmail(apiFunction);
```

生成された IAM ポリシー：

```json
{
    "Action": ["ses:SendEmail", "ses:SendRawEmail"],
    "Effect": "Allow",
    "Resource": "arn:aws:ses:ap-northeast-1:534985634165:identity/surfic.com"
}
```

**結果**: `AccessDeniedException`

**問題点の見落とし**:
- `cdk diff` では変更が表示されたが `cdk deploy` で "no changes" と言われ混乱
  - 原因: stack が environment-agnostic（`env` 未設定）のため CDK は `${AWS::Region}` という CloudFormation 疑似パラメーターを使って合成しており、`env` を設定すると hardcode ARN になる。CloudFormation 的には等価なので "no changes" になる
  - 実際にはポリシーは正しく配備されていた

---

## 試行② `resources: ['*']` → 成功

最小化テストとして全 SES リソースを許可してみた。

```ts
apiFunction.addToRolePolicy(new aws_iam.PolicyStatement({
    actions: ['ses:SendEmail', 'ses:SendRawEmail'],
    resources: ['*'],
}));
```

**結果**: HTTP 204（成功）、メール到達確認

---

## 試行③ `resources: ['identity/*']` → 失敗

`*` は広すぎるので identity のみに絞ってみた。

```ts
resources: [`arn:aws:ses:ap-northeast-1:${this.account}:identity/*`]
```

**結果**: `AccessDeniedException` 再発

---

## 試行④ `resources: ['identity/*', 'configuration-set/*']` → 成功

**根本原因の特定**: SES v2 の `SendEmail` は、ドメインに関連付けられた **configuration set** に対しても IAM 権限チェックを行う。`identity/*` だけでは configuration set への権限が不足していた。

```ts
apiFunction.addToRolePolicy(new aws_iam.PolicyStatement({
    actions: ['ses:SendEmail', 'ses:SendRawEmail'],
    resources: [
        `arn:aws:ses:ap-northeast-1:${this.account}:identity/*`,
        `arn:aws:ses:ap-northeast-1:${this.account}:configuration-set/*`,
    ],
}));
```

**結果**: HTTP 204（成功）

---

## 結論

AWS SES v2 の `SendEmail` IAM 権限は、**identity と configuration-set の両方**に対して付与する必要がある。
CDK の `EmailIdentity.grantSendEmail` は identity のみを対象にするため、configuration set が存在する場合は不十分になる。

### 最終的な CDK コード

```ts
import { aws_iam } from 'aws-cdk-lib';

apiFunction.addToRolePolicy(new aws_iam.PolicyStatement({
    actions: ['ses:SendEmail', 'ses:SendRawEmail'],
    resources: [
        `arn:aws:ses:ap-northeast-1:${this.account}:identity/*`,
        `arn:aws:ses:ap-northeast-1:${this.account}:configuration-set/*`,
    ],
}));
```

## 副産物: us-east-1 に不要なスタックが作成された

デバッグ中に `aws/bin/aws.ts` の `env` 未設定のまま `cdk deploy` を実行したため、AWS CLI デフォルトリージョン (`us-east-1`) にも `lambda360form` スタックが作成された。

削除する場合:
```bash
# us-east-1 で削除
AWS_DEFAULT_REGION=us-east-1 npx aws-cdk destroy
# または AWS Console で us-east-1 の lambda360form スタックを手動削除
```

現在の `aws/bin/aws.ts` には `env: { region: 'ap-northeast-1' }` を設定済み。

## 余談: SES identity 重複エラー

CDK で `new aws_ses.EmailIdentity(...)` を使うと、すでに手動作成済みの `surfic.com` identity と衝突して CloudFormation エラーになる。`fromEmailIdentityArn` でインポートする方法を使うこと。

```ts
// NG: すでに存在する場合エラー
new aws_ses.EmailIdentity(this, 'SesIdentity', {
    identity: aws_ses.Identity.domain('surfic.com'),
});

// OK: 既存 identity を参照するだけ
aws_ses.EmailIdentity.fromEmailIdentityArn(
    this, 'SesIdentity',
    `arn:aws:ses:${this.region}:${this.account}:identity/surfic.com`,
);
```
